> 状态：方案讨论中，未定稿

## 0. 技术路线（已对齐）

### 0.1 选定方案

**方案：端侧采集型**——坐席使用国产安卓工作机，借助厂商系统自带的"通话自动录音"功能完成录音；App 不参与拨号也不参与录音，只负责：

1. 下发任务清单 / 显示业主资料；
2. 通过 Intent 调起系统拨号盘发起通话（坐席真实号码外显）；
3. 监听通话状态变化（OFFHOOK/IDLE）；
4. 通话挂机后，根据"对端号码 + 起止时间"在系统录音目录中匹配文件；
5. 上传录音 + 通话元数据；
6. 让坐席在 App 内完成业务表单（投票表态 / 催收承诺 / 发支付链接）。

```
[Web 后台]                   [国产安卓工作机]                     [云端]
   │ 下发名单                    │  ① App 拉起系统拨号盘            │
   ├──────────────────►  App ───┤  ② 系统录音 → /Sounds/...        │
   │                             │  ③ App 扫描匹配文件               │
   │                             │  ④ 上传录音 + 元数据              │
   │                             ├──────────────────────────────────►│
   │                             │                                   │ ASR
   │                             │                                   │ ↓
   │                             │                                   │ LLM 抽取
   │                             │                                   │ ↓
   │ ◄────────────────── 业务回写（投票/承诺/支付/标签）─────────────┤
```

### 0.2 关键约束（必须满足）

| 维度 | 约束 |
|------|------|
| 坐席设备 | 必须为支持系统通话自动录音的机型：华为/荣耀（EMUI/HarmonyOS）、小米/红米（MIUI/HyperOS）、OPPO（ColorOS）、vivo（OriginOS）、魅族（Flyme）。**iPhone、Pixel、三星海外版禁用作坐席机** |
| 设备来源 | 公司统一采购工作机 + 公司名下手机卡，**不允许 BYOD**（防止业主资源被坐席带走、防止个人号码被运营商标记为骚扰号） |
| 系统设置 | "通话自动录音"开关必须开启；App 启动自检，未开启时禁用呼叫并引导设置 |
| 安卓版本 | ≥ Android 10（HarmonyOS 等同适配）；权限模型按 Scoped Storage 设计 |
| 文件采集 | App 维护"厂商→录音目录→命名规则"的适配 manifest，未识别厂商走 MediaStore 兜底 |
| 合规告知 | 坐席开场白必须包含"通话录音用于服务质量"告知，未告知通话标记为合规风险并质检告警 |

### 0.3 各厂商录音目录适配（v0.2 初步清单，落地需实测）

| 厂商 | 默认录音目录 | 文件名规则（典型） | 格式 |
|------|------------|------------------|------|
| 华为/荣耀 | `/storage/emulated/0/Sounds/CallRecord/` | `<号码>_<yyyyMMddHHmmss>.m4a` | m4a |
| 小米 | `/storage/emulated/0/MIUI/sound_recorder/call_rec/` | `通话录音_<号码>_<时间>.mp3` | mp3 |
| OPPO | `/storage/emulated/0/Recordings/Call Recordings/` | `<号码>_<时间>.amr/m4a` | amr/m4a |
| vivo | `/storage/emulated/0/记录/通话录音/` | `<号码>_<时间>.m4a` | m4a |
| 魅族 | `/storage/emulated/0/Recorder/call/` | 类似 | m4a |
| 兜底 | `MediaStore.Audio` 查询，按时间窗 ±5 分钟 + 号码模糊匹配 | — | 任意 |

**风险**：厂商可能在大版本升级时变更目录或加密，需要建立"机型适配回归测试集"和"采集失败上报"机制，每月跑一次实机回归。

### 0.4 通话事件 → 录音文件 匹配逻辑

```
1. App 监听 TelephonyCallback (Android 12+) / PhoneStateListener
2. 拿到 OFFHOOK→IDLE 转换 + 对端号码 + 起止时间戳
3. 通话结束后启动 30 秒轮询窗口（录音文件可能延迟落盘）
4. 在厂商对应目录扫描，匹配规则：
   - 文件名包含对端号码（脱敏比对，去前缀 +86/0）
   - 文件 mtime ∈ [挂机时间, 挂机时间 + 30s]
   - 文件大小 > 0 且不再增长（避免上传半成品）
5. 命中 → 拷贝到 App 私有目录 + 排队上传
6. 30 秒未命中 → 上报"录音缺失"事件，提醒坐席手动补传
```

## 1. 背景与目标

### 1.1 背景
物业公司有两类高频外呼业务，**两者都是 MVP 范围内的刚需**：

1. **业主投票邀请**：重大事项（电梯改造、维修基金动用等）需要业主表决，电话沟通时需当场确认投票意向并落库。
2. **物业费催收**：欠费业主电话沟通后，根据态度推送支付链接、约定缴费时间或转入下一步处置流程。

现状痛点：
- 通话过程无证据留存，业主事后翻供、纠纷处理困难。
- 投票/催收结果靠坐席手填，错漏多、统计滞后。
- 管理层没有质检手段，无法评估话术效果。

### 1.2 产品目标
- **可追溯**：所有外呼留下录音 + 文字稿。
- **结构化**：从通话内容自动提取关键字段（投票选项 / 缴费承诺时间 / 推托理由）。
- **闭环**：通话结束后自动触发后续动作（发支付链接、登记投票、加入再呼队列）。
- **可质检**：管理端按关键词、情绪、合规话术抽查。

### 1.3 非目标（本期不做）
- 实时通话中 AI 提示话术。
- IVR 机器人自动外呼（本期仍是人工坐席）。
- 多语种支持（仅普通话 + 主要方言）。

## 4. 功能模块

### 4.1 Web 管理后台
- **名单管理**：Excel/CSV 批量导入；字段含手机号、姓名、楼栋房号、欠费金额/投票事项 ID、历史联系摘要。
- **任务分派**：按楼栋/欠费金额/优先级分派给坐席；支持批量下发到指定工作机。
- **话术模板**：投票/催收两类，含开场白（含强制录音告知文案）、关键问句、合规话术。
- **录音与文字稿查阅**：列表 + 详情页，文字稿支持关键词跳到录音对应秒数；标记"录音缺失/匹配失败"的通话便于补救。
- **质检与报表**：通话量、接通率、投票回收率、缴费承诺转化率、平均通话时长、录音采集成功率（按机型分布）。
- **黑名单/免打扰**：业主投诉后加入；接入工信部要求的拒接清单。
- **设备/坐席管理**：注册工作机（IMEI / 设备指纹）、绑定坐席、查看每台机录音采集成功率与故障告警。

### 4.2 坐席 App（安卓工作机）

#### 4.2.1 启动自检
- 检查"系统通话自动录音"是否开启（按厂商不同读对应的设置项或引导用户进设置页）；
- 检查权限：`READ_PHONE_STATE`、`READ_CALL_LOG`、`READ_MEDIA_AUDIO` / `READ_EXTERNAL_STORAGE`、`POST_NOTIFICATIONS`、（部分厂商需要）`MANAGE_EXTERNAL_STORAGE`；
- 检查录音目录可读性（试读最近一条系统录音）；
- 任何一项不通过 → **禁用呼叫按钮** + 一键引导跳转设置。

#### 4.2.2 任务面板
- 今日待呼名单按优先级排序；
- 业主卡片：姓名、楼栋房号、本次目的（投票事项/欠费金额）、历史联系摘要 ≤ 3 条。

#### 4.2.3 一键呼叫
- 点击号码 → `Intent.ACTION_CALL` 拉起系统拨号盘并发起呼叫；
- 启动前台服务，监听 `TelephonyCallback#onCallStateChanged`；
- 通话面板浮窗显示：业主名片、计时器、业务表单（按 task.type 切换 投票 / 催收）。

#### 4.2.4 通话中操作
- **投票场景**：单选/多选投票选项、备注输入、合规提示（开场白未读则告警）。
- **催收场景**：发送支付链接（短信通过后端下发）、缴费承诺日期选择、推托原因分类、下一步处置（再呼/上门/转法务）。

#### 4.2.5 挂机后
- 进入 30 秒录音匹配窗口（详见 0.4）；
- 命中 → 拷贝到 App 私有目录排队上传；未命中 → 弹窗提示"录音未找到，请检查系统设置或手动选择文件"；
- 业务表单 30 秒内未提交 → 强提醒；
- 离线 / 弱网 → 上传任务持久化，恢复后自动续传。

#### 4.2.6 风控与合规
- 单业主每日呼叫次数上限（默认 3 次，可后台调整）；
- 外呼时段限制（默认 8:00–20:00）；
- 检测坐席尝试关闭系统录音开关 → 上报告警；
- 低电量 / 存储不足 → 提醒并暂停呼叫。

### 4.3 云端处理流水线
- **录音入仓**：App 上传分片 → 对象存储；后端校验文件完整性、声纹/单双声道分类、时长合理性。
- **ASR 转写**：调用语音识别（支持单声道分离效果有限，后续可结合声学说话人分离区分坐席/业主），输出带时间戳的文字稿。
- **结构化抽取（LLM）**：按 task.type 选择 prompt：
  - 投票：抽取业主表态（同意/反对/弃权/未明确）+ 关键理由 + 是否完成合规告知。
  - 催收：抽取缴费意愿（已承诺/推托/拒缴）+ 承诺日期 + 推托理由分类 + 是否完成合规告知。
- **校验与回写**：
  - 投票：LLM 抽取结果与坐席勾选不一致 → 进入复核队列，可选下发 H5 给业主二次确认。
  - 催收：抽取的承诺日期与坐席录入对照，不一致告警；触发短信支付链接回执监听。
  - 高风险关键词（辱骂/威胁/虚假承诺）→ 推送质检告警。

### 4.4 业主侧 H5
- **支付链接**：短信 + 链接，点击后展示欠费明细，对接微信/支付宝。
- **投票二次确认 H5**（可选启用）：短信下发投票链接，作为电子凭证补强。

## 5. 两个业务场景流程

### 5.1 场景 A：业主投票邀请
```
管理员创建投票事项 → 关联业主名单 → 分派坐席
   ↓
坐席工作机收到任务 → 启动自检通过 → 一键呼叫
   ↓
系统拨号盘发起 → 业主接听（看到坐席真实手机号）
   ↓
通话中坐席宣读议题 → 业主表态 → 坐席勾选投票选项
   ↓
挂机 → 30 秒内匹配并上传录音 → 云端 ASR + LLM 校验
   ↓ 一致：入投票库；统计实时进度
   ↓ 不一致：复核队列 / 发 H5 二次确认
```

### 5.2 场景 B：物业费催收
```
导入欠费名单 → 按优先级分派 → 坐席查看历史 → 一键呼叫
   ↓
分支：
  1. 立即缴费 → App 一键发送支付链接 → 监听支付回调
  2. 承诺缴费日期 → 记录承诺日 → 系统到期前一天自动入再呼任务
  3. 推托/拒缴 → 选择推托原因 → 转入对应处置流程
  4. 失联/空号 → 标记，触发上门核实工单
   ↓
挂机 → 录音上云 → LLM 校验承诺日期/推托原因 → 更新业主画像
```

## 7. 非功能需求

- **录音存储**：原文件保留（通常 m4a/mp3/amr），需要时统一转 16kHz 单声道 wav 给 ASR；按月归档冷存储；保留 ≥ 2 年（合规）。
- **录音采集成功率**：≥ 98%；按机型/系统版本拆分监控，掉到阈值以下自动告警。
- **ASR 准确率**：普通话 ≥ 92%，主要方言（粤/川/吴）≥ 80%。
- **抽取延迟**：通话挂机后 ≤ 120 秒返回结构化结果（含 30 秒录音匹配窗 + 上传 + ASR + LLM）。
- **合规**：
  - 开场白录音告知；
  - 业主明示拒绝立即挂断并入免打扰；
  - PII 脱敏后再进 LLM；
  - 工信部呼叫频次/时段限制（8:00–20:00）。
- **可用性**：后端与 ASR 99.9%；坐席 App 离线可用且断网可续传。
- **设备准入**：必须在白名单机型清单内，未在清单内的工作机拒绝登录。

## 8. 用户故事（INVEST 格式）

### 8.1 坐席（Agent）
- **U-A1**：作为坐席，App 启动时自动告诉我"系统通话录音"是否开启，没开就一键带我去设置页。
- **U-A2**：作为坐席，我希望打开 App 就看到今天要拨打的清单和每个业主的历史联系摘要。
- **U-A3**：作为坐席，点击呼叫直接拉起系统拨号盘拨出，挂机后 App 自动找到刚才的录音并上传，我不用手动操作文件。
- **U-A4**：作为投票场景的坐席，我希望在通话中直接勾选业主的表态选项并保存。
- **U-A5**：作为催收场景的坐席，我希望一键给业主发支付链接，并记录他承诺的缴费日期。
- **U-A6**：作为坐席，开场白漏读"通话将被录音"时，App 在挂机时提醒我。
- **U-A7**：作为坐席，录音匹配失败时 App 弹窗让我手动选择文件或重传，避免任务空白。
- **U-A8**：作为坐席，业主在通话中明确拒绝接听营销电话时，我能一键加入免打扰。

### 8.2 业务管理员（Manager）
- **U-M1**：作为管理员，我希望批量导入业主名单（Excel）并按楼栋/欠费段分派给不同坐席。
- **U-M2**：作为管理员，我希望看到每个坐席当天的接通率、平均时长、投票回收率、缴费承诺转化率。
- **U-M3**：作为管理员，我希望看到每台工作机的录音采集成功率，掉了能立刻告警。
- **U-M4**：作为管理员，我希望按业主搜索查看所有历史通话录音和文字稿，关键词高亮可跳转。
- **U-M5**：作为投票发起人，我希望投票截止前看到实时进度（已表态/同意/反对/未触达）。
- **U-M6**：作为催收负责人，我希望一键导出"承诺今日到期未缴"清单，自动加入次日再呼任务。

### 8.3 质检员（QC）
- **U-Q1**：作为质检员，我希望系统自动标记包含违规关键词、未播报录音告知的通话。
- **U-Q2**：作为质检员，我希望对单通通话打标（合格/不合格/待复核）并写质检意见。
- **U-Q3**：作为质检员，我希望抽样听完后能反向训练 LLM 抽取规则（标注样本）。

### 8.4 业主（Owner）
- **U-O1**：作为业主，我希望接到电话开场就听到"通话录音用于服务质量"的告知，知情后再继续。
- **U-O2**：作为业主，我希望收到的支付短信链接打开能看到欠费明细。
- **U-O3**：作为业主，电话中我说了"反对"，必要时希望收到一条短信复核我的投票，避免被代填。
- **U-O4**：作为业主，我希望我说"不要再打了"后真的不再被打扰。

### 8.5 系统/IT 管理员（Sys Admin）
- **U-S1**：作为 IT 管理员，我希望可以集中管理所有工作机（注册、绑定坐席、远程注销）。
- **U-S2**：作为 IT 管理员，我希望维护"机型→录音目录→命名规则"适配表，新机型一周内能加进白名单。
- **U-S3**：作为 IT 管理员，我希望坐席尝试关闭系统录音开关或卸载 App 时立即告警。
- **U-S4**：作为 IT 管理员，我希望坐席离职可一键回收工作机数据并解绑业主资源。

## 9. 待讨论 / 未决问题（v0.2 更新）

1. **工作机采购**：统一品牌（建议华为/荣耀，系统稳定 + 录音目录历史最一致）还是混合？预算与备机数量？
2. **手机卡号**：公司名下号码池统一办卡，还是每个坐席用现有公司号？涉及 SIM 持有人变更的合规流程。
3. **机型白名单首发清单**：MVP 先支持哪 3–5 个机型？取决于现场实测结果。
4. **录音目录适配回归**：每月一次的实机回归测试由谁来跑？是否要建一个"小规模实机测试间"？
5. **录音文件加密**：部分机型可能开启录音加密（极少见），App 需有降级策略（提示坐席改设置）。
6. **投票电子凭证**：仅录音是否够（部分地方法规要求双重凭证），是否所有投票通话都强制 H5 二次确认？
7. **支付通道**：自建小程序商户号 vs 接入物业现有缴费系统。
8. **方言识别**：项目所在地方言比例多高？是否需要专门方言模型？
9. **数据合规**：ASR/LLM 选型（讯飞/阿里/火山/腾讯均在境内），录音和文字稿存储区域。
10. **离职与资源交接**：工作机回收 SOP；业主联系数据如何脱离离职坐席并接给继任者；是否需要把坐席手机号也回收（防止业主继续打过来）。
11. **应用市场上架**：是否走应用商店分发，还是 MDM 内部直装？走商店时 `READ_CALL_LOG` 等敏感权限审核风险。
12. **坐席激励 / 质检挂钩**：是否把"录音采集成功率"纳入 KPI？

## 10. 建议的 MVP 切分（v0.2，双场景刚需版）

| 阶段 | 周期 | 内容 |
|------|------|------|
| Sprint 0 | 1–2 周 | 机型实测 + 录音目录适配 manifest + 录音匹配 PoC（单台华为/小米/OPPO 各一台） |
| MVP（6–8 周）| 双场景同时上 | Web 后台（名单/任务/录音查询）；安卓 App（自检+拨号+录音采集+上传+投票/催收两套表单）；后端 API + 对象存储 + ASR；催收支付链接闭环；投票录音留证 |
| V1（+3–4 周）| LLM 自动化 | LLM 自动抽取投票意向 / 催收承诺 + 校验告警 + 投票 H5 二次确认（可配置开关） |
| V2（+4 周）| 智能化与质检 | 质检规则引擎、违规关键词告警、报表中心、机型采集成功率监控大盘 |
| V3 | 机器人外呼 | IVR 智能预呼，过滤空号/拒接，再转人工 |

**Sprint 0 的产出门槛**：在 3 种机型上完成"挂机后 30 秒内 ≥ 95% 概率匹配到正确录音文件并上传成功"，否则不进 MVP 编码阶段。
