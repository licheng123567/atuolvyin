# MVP 实现差距分析（GAP_ANALYSIS）

> 生成日期：2026-05-05
> 依据：3 个 Explore agent 并行调研 frontend/src/ 真实 React 代码 vs PRD §21 详细页面规范
> 目的：揭示 UI_GAPS.md "73 个 P0 ✅" 与 React 实际实现的真实差距，为后续 Sprint 排期提供决策依据

---

## 1. 关键结论

**UI_GAPS.md 严重高估了实际进度。** 它统计的是 `ui/*.html` 静态设计原型——而项目真正的实现层是 `frontend/src/`（React + Refine.dev）。

经过对 11 个角色（含 7 个 PRD 角色 + legal/workorder/project_manager）的逐页核查：

| 维度 | 数字 |
|------|------|
| PRD §21 P0 页面总数 | ~50 个（按角色加总）|
| **真实 React 完整实现** | **约 5 个**（10%）|
| **React 基础实现（缺关键功能）** | **约 8 个**（16%）|
| **React 完全缺失** | **约 37 个**（74%）|
| 完全 0 实现的角色 | 5 个（provider_admin / legal / workorder / project_manager × 2） |

## 4. 重新规划：原批 2/批 3 已不适用

之前的规划（批 2 = Android 推送 + 结算；批 3 = 工单 + 法务 + 二维码 + 周报 + Excel 导入）建立在"73 P0 已实现"假设上。**这个假设错了**——admin 看板首页、公海管理、结算管理这些 PRD 描述详细的核心页面 React 端**根本没有**。如果不补齐这些，仅做新功能（推送/结算 API），上线后用户登入只能看到"案件列表"一个页面，体验非常差。

## 5. 推荐分批方案

按"对 MVP 真正可用的贡献度"分组：

### 批 2A：CRM 闭环核心（6-8 天）— **必做，先做**

让 admin 和 agent 至少能完整跑一个真实业务流程：

1. admin 管理看板首页（KPI + 配额预警 + 排名）
2. admin 公海管理 + 案件看板视图（拖拽阶段流转）
3. admin 案件详情补齐（活动时间线、分配/转法务/建工单按钮）
4. agent 实时通话工作台补齐（**风控 L1/L2 banner 渲染** + 建工单/发码/转接按钮）—— **5a 风控的"最后一公里"**
5. supervisor 质检复核工作台（录音播放器 + 改 AI 标签 + 打标）
6. agent_external 脱敏前端守卫（堵漏洞）

### 批 2B：商业化必备（4-5 天）—— 必做

7. admin 结算管理 + 付款凭证（PRD §19 MVP 范围核心）
8. platform_ops 服务商列表 + 审核 + 详情（合作模式必备）
9. platform_ops 续费/停用/试用跟进
10. Android 推送 + 录音降级（之前的 Sprint 7）

### 批 3：完整角色补全（6-8 天）—— 时间允许时做

11. legal / workorder / project_manager 三角色 React 化（从 HTML 原型搬过来）
12. provider_admin 4 P0 页（服务商工作台）
13. platform_super 系统健康/审计日志/成本看板/套餐配置

### 批 4：增量增强（推迟）

14. 之前批 3 的小功能（D 周报 / 二维码 / Excel 导入）
15. PRD P1 stub 页面填充
16. 多维报表 / 客户跟进记录等

---

## 6. 核心建议

**当前 MVP 不该急着上 Android 推送和结算**，因为基座不全。先补 CRM 闭环（批 2A）让 admin/agent/supervisor 三角色能跑通日常流程，再做商业化（批 2B）。

实际推进顺序建议：
```
批 2A（CRM 闭环）→ Checkpoint → 批 2B（商业化必备）→ Checkpoint → 批 3（角色补全）→ MVP v1.0
```

每一批完结后做一次浏览器视觉验收，避免再次出现"以为做完了实际差距很大"。

如时间紧张，可以**只做批 2A + 2B**（10-13 天）作为 MVP v1.0 候选，批 3 留 v1.1。
