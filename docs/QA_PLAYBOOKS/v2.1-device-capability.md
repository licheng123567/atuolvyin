## Part C — Android 真机 / Emulator 手测

### C.1 首次安装 Onboarding 4 步（Task 5）

```bash
cd poc/android
./gradlew :app:installDebug
adb shell pm clear com.autoluyin.demo  # 重置 onboarding 状态
adb shell am start -n com.autoluyin.demo/.MainActivity
```

#### Step 1 — 权限授予

- [ ] 7 项权限逐条显示中文标题 + 解释：CALL_PHONE / RECORD_AUDIO / READ_MEDIA_AUDIO（或 READ_EXTERNAL_STORAGE）/ READ_PHONE_STATE / READ_CALL_LOG / READ_CONTACTS / POST_NOTIFICATIONS
- [ ] 点 "授予" 弹系统权限对话框
- [ ] 拒绝任何一项后页面置灰 "下一步"，可二次发起
- [ ] 全部授予后 "下一步" 高亮可点

#### Step 2 — 后端地址配置

- [ ] 默认填 `http://192.168.31.242:18000`（local-dev）或 build flavor 注入的生产地址
- [ ] 输入合法 URL 后绿色对勾
- [ ] 点 "测试连接" 走 `/health`，2s 内返回 200 显示 "连接正常"
- [ ] 失败给红色提示 + 不阻塞下一步（用户可保存后修）

#### Step 3 — 录音设置

- [ ] 顶部显示本机 ROM 厂商标签（"MIUI 设备" / "EMUI 设备" / "海外/AOSP 设备"），由 `derive_rom_family(Build.MANUFACTURER)` 决定
- [ ] 文案 "请到系统设置 → 通话 → 开启自动录音"
- [ ] 提供 "打开通话设置" 按钮（深链 `Intent(Settings.ACTION_SOUND_SETTINGS)` 或厂商专属）
- [ ] checkbox "我已开启通话自动录音" 必勾才能下一步
- [ ] 勾选后写 SharedPreferences `KEY_RECORDING_TOGGLE_SELF_REPORTED = true`

#### Step 4 — 完成

- [ ] 显示总结 + "进入应用" 按钮
- [ ] 点击触发 `AppConfig.markOnboardingDone()`，跳 MainActivity
- [ ] 下次冷启动不再进 Onboarding（直接 MainActivity）

### C.2 capability 持久化 + 录音扫描失败回写（Task 4）

#### 场景 1 — 首次自检

- [ ] 登录后 MainActivity 自动触发 `DeviceCapabilityProbe.runSelfCheck()`
- [ ] adb logcat 看到 `DeviceCapabilityProbe: probing → capability=realtime, rom=Xiaomi MIUI 10`
- [ ] `adb shell run-as com.autoluyin.demo cat shared_prefs/app_config.xml | grep capability` 看到对应值

#### 场景 2 — 通话挂断扫不到录音

- [ ] 在 Android 13 + MIUI 14 机器上拨打 demo 案件
- [ ] 通话挂断后 logcat `CallWatcherService: scan files in /storage/.../call/... not found`
- [ ] 触发 `AppConfig.markRecordingScanFailed(true)`
- [ ] 重新启动 App → 下一次 self-check 上报 `last_recording_scan_failed=true`
- [ ] 后端响应 `capability = "incompatible"` （即使矩阵命中 post_upload 也被覆盖）
- [ ] App 持久化新 capability，banner 立即变红

#### 场景 3 — 录音扫描成功

- [ ] 同一台机器再次拨打，录音被成功匹配 + 上传 200
- [ ] `AppConfig.markRecordingScanFailed(false)` 清标志
- [ ] 下一次 self-check 上报 `last_recording_scan_failed=false`，capability 回到矩阵判定

> 注：runtime 降级是 **单向收紧**，扫到一次失败 → incompatible；再扫成功只是 **解除标志**，下一次自检按矩阵重判（避免反复跳变骚扰用户）。

### C.3 三套机型 banner 颜色（Task 6）

| 机型 | 期望 capability | banner | 可关闭 | 详情链接 |
|---|---|---|---|---|
| Android 9 + MIUI 10 | `realtime` | 绿 | 是（X 按钮）| 否 |
| Android 13 + MIUI 14 | `post_upload` | 橙 | 否 | 是（点击跳 profile）|
| Pixel 8 Android 14 | `incompatible` | 红粗体 | 否 | 否 |

- [ ] 三机分别登录后 `/app/home` 看到对应颜色 banner
- [ ] banner 文案与 capability matrix 服务返回的 `guidance` 一致
- [ ] 关闭后（仅绿 banner 可关）刷新页面再次出现（不持久化关闭状态，保持每次进 home 都可见）

### C.4 incompatible 设备拨号 confirm（Task 6）

- [ ] Pixel 8 登录后进 `/app/cases/1`
- [ ] 点 "发起外呼"
- [ ] 弹 `window.confirm` 内容含 "无法保存通话录音，仍要继续吗？"
- [ ] 点 "确定" → 走 `AndroidBridge.dialCase(caseId)`，正常拨号但不期望录音
- [ ] 点 "取消" → 拨号被中止，留在案件详情页
- [ ] realtime / post_upload 设备拨号 **不弹** confirm，直接拨

## 已知差异 / 待优化

| # | 项 | 影响 | 待 |
|---|----|------|---|
| 1 | window.confirm 而非 shadcn AlertDialog | 视觉品牌不一致 | v2.x 统一 UI 组件库后替换 |
| 2 | supervisor 看本租户全部 agent（未建模 subordinates 关系）| 范围过宽 | v2.2 引入 `supervisor_agent_relation` 表 |
| 3 | platform_superadmin 无 sidebar 入口（仅可直接访问 URL） | 易用性 | 按产品需求决定是否加入口 |
| 4 | live-wall 前端 join（admin/agent-devices?page_size=200）| 性能瓶颈 | 租户 > 200 坐席时改后端 join + WebSocket 推 |
| 5 | onboarding step 3 用户可无脑勾确认 | 准确性 | RecordingScanner 失败时 runtime 自动降级兜底 |

| 7 | scan_failed 一次就降级，无重试容忍 | 偶发漏抓直接掉 incompatible | 后续加 "连续 N 次失败再降级" 阈值 |

## 附：故障排查

### Part A 某测试失败

1. 看 stacktrace 是否 testcontainers Postgres 启动超时 → 重跑 `pytest --timeout=120`
2. capability 计算结果与预期不符 → 在 `tests/services/test_device_capability_matrix.py` 加 print debug，对照 §8.4.2 矩阵
3. `device_capability_log` 写入缺字段 → 看 alembic 是否漏跑迁移：`alembic current` 应到最新 head

### Part B banner 元素找不到

- 看 frontend dev server 是否起 5173；浏览器手开 http://localhost:5173/app/home（带 fake JWT 跳登录就重新 init）
- 看 `frontend/src/hooks/useCapability.ts`（或对应 hook）是否正确调用 `window.AndroidBridge?.getCapability()`
- 看 `frontend/src/components/CapabilityBanner.tsx` className 是否包含 `cap-banner` / `cap-banner-orange` / `cap-banner-red`

### Part C onboarding 不出现

- 确认 SharedPreferences 已清：`adb shell run-as com.autoluyin.demo rm -rf shared_prefs/app_config.xml`
- 或 `pm clear com.autoluyin.demo` 一刀清

### Part C 三机 capability 结果与预期不符

- adb logcat 看 `DeviceCapabilityProbe` tag 输出的 manufacturer / model / android_version 是否与机器实际一致
- 如某真机被识别为 unknown（如 nubia / lenovo / zte），属矩阵未覆盖项，记入已知差异 #6 待 v2.2 扩矩阵

### Part D 列表为空

- 确认有坐席跑过自检：用 demo 账号在 Android 真机或 emulator 走一次登录
- 看后端日志：self-check 是否成功写 `device_capability_log`
- 看是否 capability 筛选过窄：清筛选后再看
